# TickTick — Core Product Features & Data Model

**App:** `com.ticktick.task` (TickTick)
**version_code:** `8081`  **versionName:** `8.0.8.1`

Decompiled trees analyzed (read-only):
- jadx Java: `/home/user/apk-workspace/out/jadx/sources/`
- apktool smali: `/home/user/apk-workspace/out/apktool/` (dex map: `smali`=classes.dex,
  `smali_classes2`=classes2.dex … `smali_classes5`=classes5.dex)

**Obfuscation posture.** `com/ticktick/task/*` *logical* class names are mostly
preserved (entities, DAOs, services, API interfaces, activities). Renamed
("rotated") code is the helper layer that R8 hoisted to top-level single-letter
packages: `yb/`, `kc/`, `zb/`, `ec/`, `dc/`, `ic/`, `bc/`, `ac/`, `eb/`, `q9/`,
`vb/` etc. Within preserved classes the *method/field* names are still rotated
(GreenDAO entities are an exception — their getters/setters survive). So the
rotation-stable anchors below are string literals or kept super/interface refs,
never member names. Native code: `lib/arm64-v8a/libnative_parser.so` (the NL
date/quick-add parser), `libmmkv.so` (Tencent MMKV key-value storage).

The authoritative obfuscated→logical name mapping and regex anchors live in
`signatures/com.ticktick.task/signatures.yaml` + the map; this doc is the
what-it-does narrative and references code by `file:line`.

## 1. Tasks & local persistence (GreenDAO)

**Persistence engine: GreenDAO** (not Room). `org.greenrobot.greendao.*` is the
ORM; the DB file is a plain SQLite database whose tables are generated by the
`*Dao` classes under `com/ticktick/task/greendao/`.

- **`DaoMaster`** (`greendao/DaoMaster.java`) — `SCHEMA_VERSION = 216`
  (`DaoMaster.java:11`). `createAllTables`/`dropAllTables` call `createTable` on
  **102 DAOs** (`grep -c Dao.createTable` = 102). `OpenHelper`/`DevOpenHelper`
  wrap the schema-version upgrade. Extends `org.greenrobot.greendao.b`.
- **`DaoSession`** (`greendao/DaoSession.java`) — the central registry: holds and
  lazily builds all 102 `*Dao` instances (102 `get…Dao()` getters). Extends
  `org.greenrobot.greendao.c` (= `Lorg/greenrobot/greendao/c;`, the stable super
  anchor; the class has **no string literals**). This is the object every entity
  caches in its transient `daoSession` field for lazy relation resolution.
- **DAO wrappers** (`com/ticktick/task/dao/*DaoWrapper.java`) — app-level repos
  that wrap each generated `*Dao` with query helpers (e.g. `Task2DaoWrapper`,
  `HabitDaoWrapper`). These are the layer the `service/` classes call.

## 2. Sync engine

TickTick has a **cross-platform sync core** (kotlin-multiplatform style,
`common_release` module) under `com/ticktick/task/sync/`, plus an Android-side
wrapper. The central class is the sync-core `SyncService`.

- **`com.ticktick.task.sync.sync.SyncService`** (`sync/sync/SyncService.java`,
  dex = classes4) — the sync engine entry. Constructed with a `w9.e` syncResult
  sink. Entry method **`doSync(int type)`**. Flow (from the kept `@Metadata`
  signatures + method bodies):
  1. **pull phase** — `pull()`, `pullColumns()`, `pullTasksOfOpenedProjects()`,
     `pullUserStatus()`, `pullUserLimit()`, `pullNotificationCount()`,
     `pullOtherData()`, `handleRemoteSyncCheck(List<SyncCheckInfo>)`,
     `handleSyncStatus()`. Server delta is keyed off a **check point**
     (`getLastSyncedVersion()`, `saveCheckPoint(long)`) — i.e. incremental sync
     via `batch/check/{point}`.
  2. **commit phase** — one `commit*()` per entity kind, each draining locally
     `_status`-dirty rows and POSTing a batch: `commitTask`, `commitTaskParent`,
     `commitAddAndUpdateProject`, `commitDeleteProject`, `commitMoveProject`,
     `commitMoveColumn`, `commitColumns`, `commitTags`, `commitFilter`,
     `commitProjectGroup`, `commitDelayReminders`, `commitPomodoro`,
     `commitTiming`, `commitAssignTask`, `commitRestore`, `commitDeleteForever`,
     `commitSortOrderTask`, `commitOrderByType`/`commitOrderByTypeV3`.
  3. **trash** — `tryToClearTrash()`.
  Each commit delegates to a **BatchHandler** (held as fields:
  `taskBatchHandler`, `tagBatchHandler`, `projectBatchHandler`,
  `pomoBatchHandler`, `timingBatchHandler`, `kanbanBatchHandler`,
  `taskParentHandler`, `restoreHandler`, `deleteForeverHandler`, …) which builds
  the `Sync*Bean`, calls the API, and merges the result back.
  Unique-string anchor: `"commitPomodoro brief error conflict  = "`
  (`SyncService.java:486`; verified 1 hit in smali). Kept TAG string
  `"SyncService"` is also present but is shared with the Android wrapper.
- **BatchHandlers** (`com/ticktick/task/sync/sync/handler/*BatchHandler.java`) —
  one per entity: `TaskBatchHandler`, `TagBatchHandler` (with merge-with-server),
  `ProjectBatchHandler`, `FilterBatchHandler`, `PomoBatchHandler`,
  `TimingBatchHandler`, `KanbanBatchHandler`, `TaskParentBatchHandler`,
  `OrderBatchHandler` family, `ErrorBatchHandler`. The base is `BatchHandler`.
- **`BatchApiInterface`** (`com/ticktick/task/network/api/BatchApiInterface.java`,
  dex = classes4) — the Retrofit (kotlin) interface the handlers call; methods
  return `x7.a<…>` (TickTick's call wrapper). Endpoints (from kept path literals
  in `@o`/`@f` annotations):
  - `GET  api/v2/batch/check/{point}` → `batchCheck` (incremental delta)
  - `POST api/v2/batch/task` → `batchUpdateTasks(SyncTaskBean)`
  - `POST api/v2/batch/tag` → `batchUpdateTags(SyncTagBean)`
  - `POST api/v2/batch/project` → `batchUpdateProjects(SyncProjectBean)`
  - `POST api/v2/batch/projectGroup` → `batchUpdateProjectGroups`
  - `POST api/v2/batch/filter` → `batchUpdateFilters(SyncFilterBean)`
  - `POST api/v2/batch/taskParent` → `batchUpdateTaskParent(List<TaskParent>)`
  - `POST api/v2/batch/taskOrder`, `…/taskProjectSortOrder`, `…/taskProject`
    (move projects), `…/pomodoro`, `…/pomodoro/timing`,
    `…/calendar/bind/events/batch`, restore/delete-forever, assignees.
  Anchor: literal `"api/v2/batch/task"` (5 hits in smali — the family of
  `batch/task*` paths).
- **Android-side trigger:** `com.ticktick.task.helper.TimerSyncHelper`
  (`helper/TimerSyncHelper.java:115,138`) shows the call pattern used everywhere:
  `((BatchApiInterface) new vd.e(getApiDomain(), false).f33309c).batchUpdatePomodoros(...).d()`
  → handler `handleResult(...)`. `vd.e` is the obfuscated API-builder/retrofit
  factory.

**Sync model summary:** dirty-row tracking via per-entity `_status`/`SyncStatus`
table → per-kind batch POST → server returns id2etag + conflicts → handler
merges (updates SID/ETAG, clears `_status`). Incremental download via a
monotonic check-point. Readable: API interfaces & handlers keep logical names;
the builder/retrofit plumbing (`vd.*`, `x7.a`, `w9.e`) is obfuscated.

## 3. Focus / Pomodoro / Stopwatch

Two long-running foreground services drive two timer modes; both share a
**state-machine** design. The state machine itself is obfuscated into `yb/`
(decompiled `// compiled from: Pomodoro*.kt`).

### State machine (`yb/`, dex = classes.dex / `smali`)

- **`yb.c`** = **`PomodoroState`** interface (`yb/c.java`, "compiled from:
  PomodoroState.kt"). Methods (kept, framework-typed): `getTag()`,
  `isWorkFinish()`, `isRelaxFinish()`, `isInit()`, `isValid()`, `c()`, `e()`,
  `f()`, `i()`, `j()`, `k()`. Anchor: the kept abstract sig
  `isRelaxFinish()Z` / `isWorkFinish()Z` together with `getTag()Ljava/lang/String;`.

  (`b`/`C0505d`/`e` = WORK / breaks, `f` abstract finish-state with a nested
  `CountDownTimer`, `g` etc.). It maps states to wear constants
  `WearConstant.FOCUS_STATE_WORK / _SHORT_BREAK / _LONG_BREAK / _WORK_FINISH /
  _RELAX_FINISH` (`yb/d.java:228,328,488,613,778`). Transition verbs appear as

## 6. Quick-add / natural-language parsing

The NL parser that turns "buy milk tomorrow 5pm !!!" into a structured task with
due date / repeat / priority is **native C++** in `libnative_parser.so`.

- **`com.ticktick.task.utils.TitleParserLib`**
  (`utils/TitleParserLib.java`, dex = classes4) — the JNI binding.
  `static { System.loadLibrary("native_parser"); }` (anchor:
  `const-string/jumbo "native_parser"`). Native methods:
  - `String parseTitle(String text, String tz, boolean isPro, String language)`
    — the main entry (JNI symbol `Java_com_ticktick_task_utils_TitleParserLib_parseTitle`).
  - `setCustomizeHour(...)`, `setDefaultStartDay(int)`, `setDefaultTimeSecond(long)`,
    `setHolidayDay(String)`, `setJapanHolidayDay(String)`,
    `setIs24DateFormat(bool)`, `setIsUsOrUkDateFormat(bool)` — locale/holiday
    config pushed into the parser.
- **`com.ticktick.task.utils.TitleParser`** (`utils/TitleParser.java`,
  dex = classes4) — the Java facade. `parseTitle(String, Date, boolean)`
  (`TitleParser.java:172`) builds locale args and calls
  `getTitleParser().parseTitle(sub, tz, isPro, language)` (`:219`), returning a
  `ParserDueDate`. Contains locale literals (`"m/d"`, `"d/m"`, `"remind me"`,
  `"提醒我"`) as anchors. Callers: `eb/g0.java:53`, `eb/p.java:1272`
  (quick-add input pipeline).
- **Native internals** (from `_Z…` mangled symbols in the `.so`, useful for
  future native work): C++ classes `NLPNativeTitleParser` (`parseTitle`,
  `matchLunar`), `NLPRecognizeDate` (start/end cal, duration, reminder, repeat
  recog strings), `NLPParserDueDate`, `NLPRepeatResult`, `StringRRule`,
  `NLPTickTickDuration`. Free fn `parserTitle(const wstring&, …)`. The parser
  works in `wchar_t` strings and returns recognized spans + a serialized due/
  repeat/reminder result that `ParserDueDate` deserializes. (Note this native NL
  parser is separate from the *AI* voice/text parser in §5.)
- **Quick-add UI:** `com/ticktick/task/quickadd/QuickAddActivity.java` +
  obfuscated controllers (`quickadd/a–x`, `quickadd/priority/`,
  `quickadd/defaults/`, `quickadd/list/`).

Readable: `TitleParserLib`/`TitleParser` keep names; everything below is native.
